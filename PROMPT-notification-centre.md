## 3. WHAT TO BUILD — THE NOTIFICATION CENTRE UI

### 3A. Customer Portal — Account → Notification Centre

Create a Blade view at `resources/views/quicksms/account/notification-centre.blade.php` using `layouts/quicksms.blade.php`.

**Tabs/Sections:**

1. **Notifications Feed** — Real-time list of notifications from the `POST /api/notifications/` endpoint
   - Show severity badge (critical=red, warning=amber, info=blue) using existing Fillow status badge patterns
   - Show title, body, timestamp, category chip
   - Action buttons: Mark Read, Dismiss
   - "Mark All Read" bulk action
   - Filter by: category (billing, messaging, compliance, security, system, campaign), severity, read/unread
   - Empty state: "No notifications yet"

2. **Alert Rules** — CRUD interface for customer alert rules from `/api/v1/alerts/rules`
   - List existing rules with: trigger_key (human-readable label), channels, frequency, enabled toggle
   - "Add Rule" form — category dropdown, trigger_key dropdown (from config), condition operator/value, channels checkboxes, frequency dropdown
   - Edit/delete existing rules
   - System defaults shown as read-only (is_system_default=true) with option to override

3. **Alert History** — Read-only log from `/api/v1/alerts/history`
   - Table: timestamp, trigger_key, severity badge, title, status (dispatched/delivered/failed/suppressed), channels
   - Summary stats from `/api/v1/alerts/history/summary`
   - Filter by: category, severity, status, date range

4. **Preferences** — Category-level muting from `/api/v1/alerts/preferences`
   - Per-category: toggle mute, select channels, set mute-until date
   - Categories: billing, messaging, compliance, security, system, campaign

5. **Channel Settings** — Channel config from `/api/v1/alerts/channels`
   - Email: enabled toggle (email address from user profile)
   - Webhook: URL input, HMAC secret, test button
   - Slack: webhook URL input, test button
   - Teams: webhook URL input, test button
   - SMS: phone number input, enabled toggle

### 3B. Admin Console — Management → Notification Centre

Create a Blade view at `resources/views/admin/management/notification-centre.blade.php` using `layouts/admin.blade.php`.

**Tabs/Sections:**

1. **Admin Notifications Feed** — From `GET /api/notifications/` (admin routes)
   - Same UX as customer but with admin-specific categories (fraud, platform_health, customer_risk, commercial, compliance_legal)
   - Additional action: "Resolve" button (marks as resolved with timestamp)
   - Filter by admin categories

2. **Alert Rules Management** — From `/api/alerts/rules`
   - View and manage system default rules
   - Create admin-specific rules
   - Toggle rules on/off
   - Edit channels, frequency, cooldown

3. **Alert Dashboard** — From `/api/alerts/dashboard`
   - Summary cards: total alerts today, by severity, by category
   - Recent alert history feed

4. **Alert History** — From `/api/alerts/history`
   - Full history table with all columns
   - Filter by trigger_key, severity, status, tenant_id, date range

### Data flow for reference

```
User action (e.g. retention change)
  → Controller updates DB
  → safeDispatch() fires event (e.g. AccountSecuritySettingChanged)
  → AlertEventSubscriber catches it
  → Dispatches EvaluateAlertEventJob to 'alerts' queue
  → AlertEvaluatorService.evaluate()
    → Finds matching AlertRule(s) by trigger_key + tenant_id
    → Checks condition (event-type always passes)
    → Checks cooldown via AlertFrequencyService
    → If instant: AlertDispatcherService.dispatch() → CreateInAppNotificationJob + channel jobs
    → If batched: adds to NotificationBatch for later DispatchBatchedAlertsJob
    → Records AlertHistory (immutable)
  → CreateInAppNotificationJob creates Notification or AdminNotification record
  → Notification Centre UI reads Notification records via /api/notifications/
```
