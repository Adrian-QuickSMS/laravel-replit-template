  - partner() belongsTo relation on app/Models/Account.php.

WHAT IT DOES NOT ADD:
  - No controllers, routes, views, middleware, services, jobs, seeders.
  - No changes to existing RLS policies on any other table.
  - No GRANT statements on the partners table — these intentionally land in 
    PR-4 alongside SetPartnerContext middleware. Until then, partners is 
    correctly unreachable from any non-superuser role.
  - No partner data — partners table stays empty; partner_id NULL on every 
    existing accounts row.

WHY: foundation for later PRs (tenant-context helpers, RLS hierarchy, partner 
middleware, partner portal). Today's behaviour for every existing customer 
and admin is unchanged.

MODEL SEMANTICS (so test failures are easy to interpret):
  - Partner status defaults to 'active' (DB column default).
  - Partner $fillable = [legal_name, trading_name, contract_type, currency, 
    owner_account_id, created_by, updated_by]. Status is NOT fillable; 
    transitions go through Partner::transitionTo().
  - created_by and updated_by are nullable strings.
  - status check constraint allows ('active', 'suspended', 'closed').

PRODUCTION MIGRATION NOTE (informational only — do not act on this):
  Adding the partner_id column is metadata-only and instant. Adding the FK 
  triggers a validation scan (trivial — every existing row has NULL). Adding 
  the index is fast on a 100% NULL column. Inline index is safe for this 
  PR. For a future migration that backfills partner_id at scale, prefer 
  CREATE INDEX CONCURRENTLY in a follow-up.
