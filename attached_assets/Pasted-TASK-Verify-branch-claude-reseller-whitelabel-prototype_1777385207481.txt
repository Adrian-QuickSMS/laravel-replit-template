This step proves the partners_self_access USING clause actually filters rows.

CRITICAL: must run inside an explicit transaction. `SET LOCAL ROLE` only

own implicit transaction and SET LOCAL is a no-op.

S6.1  Run the entire block in one psql invocation (single transaction):

      psql "$DATABASE_URL" <<'SQL'
      \set ON_ERROR_STOP on
      BEGIN;
      INSERT INTO partners (legal_name, currency) 
        VALUES ('RLS_ENFORCEMENT_TEST', 'GBP') 
        RETURNING id \gset
      
      -- Without partner context — fail-closed expected (0 rows)
      SET LOCAL ROLE portal_rw;
      SELECT COUNT(*) AS without_context FROM partners;
      
      -- With matching partner context — should see the row (1)
      SELECT set_config('app.current_partner_id', :'id', true);
      SELECT COUNT(*) AS with_correct_context FROM partners;
      
      -- With wrong partner context — fail-closed expected (0)
      SELECT set_config('app.current_partner_id', '00000000-0000-0000-0000-000000000000', true);
      SELECT COUNT(*) AS with_wrong_context FROM partners;
      
      RESET ROLE;
      ROLLBACK;
      SQL

        with_correct_context  = 1
        with_wrong_context    = 0
